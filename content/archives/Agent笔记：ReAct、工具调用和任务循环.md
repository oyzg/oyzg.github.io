Agent 这个词很容易被用得很泛。只要模型能调用工具，就有人叫 Agent；只要有多轮对话，也有人叫 Agent。为了不混在一起，我先按最朴素的理解记：Agent 是一个能围绕目标，反复观察环境、决定动作、执行工具、读取反馈并继续推进任务的系统。

它不只是一次性回答问题，而是有任务循环。

## 从LLM到Agent

普通 LLM 调用大概是：

```text
prompt -> model -> answer
```

Agent 更像：

```text
goal -> plan -> action -> observation -> update state -> next action -> final result
```

差别在于 Agent 把模型放进了一个外部执行循环里。模型不只是生成文本，还要决定下一步做什么。下一步可能是搜索网页、读文件、运行命令、写代码、调用数据库、发请求，或者向用户确认。

所以 Agent 能力不只取决于模型，还取决于外壳系统：

- 有哪些工具。
- 工具返回什么格式。
- 状态怎么保存。
- 错误怎么处理。
- 权限怎么限制。
- 什么时候停止。

这也是我现在看 Agent 系统时更关注 harness 的原因。模型是大脑，但真正能不能完成任务，很多时候取决于循环设计。

## ReAct

ReAct 是 2022 年的工作，题目是 *ReAct: Synergizing Reasoning and Acting in Language Models*。它把 reasoning 和 acting 放到一起，让模型在推理过程中穿插外部动作。

典型模式是：

```text
Thought: 我需要查某个信息
Action: Search(...)
Observation: 搜索结果...
Thought: 现在还缺少另一个信息
Action: Lookup(...)
Observation: ...
Answer: ...
```

这个结构看起来简单，但影响很大。它让模型的中间推理不只是内部文本，而是可以驱动外部工具。工具返回的 observation 又反过来改变模型下一步判断。

ReAct 对我最重要的启发是：Agent 的推理不是一口气想完，而是边做边修正。很多任务只有执行后才知道下一步该怎么走。

## 工具调用

Tool use 是 Agent 的基础能力。没有工具，模型只能在参数和上下文里回答；有工具后，模型可以访问外部世界。

常见工具包括：

- 搜索工具：查实时信息。
- 文件工具：读写本地仓库。
- Shell 工具：运行测试、构建、脚本。
- 浏览器工具：打开网页、点击、截图。
- 数据库工具：查询业务数据。
- API 工具：调用外部服务。

工具调用的关键不只是“能调用”，还要让模型知道：

- 什么时候该调用。
- 参数怎么填。
- 调用失败怎么处理。
- 返回结果怎么解释。
- 哪些操作需要用户确认。

工具描述越清楚，模型越容易正确调用。工具返回越结构化，后续推理越稳定。反过来，如果工具输出一大堆噪声，Agent 很容易误判。

## Observation

Observation 是工具执行后的反馈。很多 Agent 系统失败，不是因为动作选择错，而是 observation 处理错。

比如执行命令后：

- 命令成功还是失败。
- 失败是语法错误、权限问题、网络问题，还是测试断言失败。
- 输出里哪些信息是关键。
- 后续是否需要重试。

模型需要从 observation 中提取状态变化。如果 observation 太长，关键错误被淹没；如果 observation 太短，又缺少修复依据。

所以 Agent 的工具层经常需要对输出做整理。例如测试失败时只保留失败用例、错误栈和相关日志。网页搜索时只保留标题、摘要和链接。数据库查询时返回结构化 JSON。

## 状态管理

Agent 任务通常不是一步完成的，所以需要状态。

状态可以包括：

- 当前目标。
- 已完成步骤。
- 待办事项。
- 已调用工具和结果。
- 中间结论。
- 失败原因。
- 用户约束。

如果没有状态管理，Agent 很容易重复做同一件事，或者忘记用户刚刚给的限制。

简单任务可以把状态放在上下文里。长任务就需要外部状态，例如任务计划、scratchpad、数据库、文件、memory 系统。

我觉得状态管理是 Agent 和普通 chatbot 的分界线之一。普通聊天只需要记住对话，Agent 需要记住任务进度。

## Planner和Executor

很多 Agent 系统会拆成 planner 和 executor。

Planner 负责把目标拆成步骤。Executor 负责执行具体动作。

这种拆分的好处是清晰。规划时先不急着调用工具，先确定大方向；执行时专注当前步骤，拿到反馈后再更新计划。

但也不能过度规划。真实任务经常执行到一半才发现前提错了。太长的计划容易变成形式主义。更实用的方式是短计划 + 反馈修正：

```text
先列出接下来2-4步 -> 执行一步 -> 观察结果 -> 调整计划
```

这和写代码很像。先理解问题，做一个小改动，跑测试，看反馈，再继续。

## 失败重试

Agent 必须面对失败。工具可能报错，网页可能打不开，测试可能不通过，文件可能不存在，模型也可能误解任务。

失败处理可以分几类：

- 参数错误：修正参数后重试。
- 信息不足：追加查询或询问用户。
- 权限不足：停止并请求确认。
- 路径错误：重新搜索文件或资源。
- 结果矛盾：回到上一步重新验证。
- 多次失败：降低目标或给出失败原因。

好的 Agent 不应该无限重试。需要有 retry budget 和停止条件。否则它会在错误路径上越走越远。

## Memory

Memory 不是所有 Agent 都必须有，但长期 Agent 离不开它。

短任务里，上下文就是 memory。跨会话任务里，系统需要记住用户偏好、项目经验、常用命令、历史错误和修复方式。

不过 memory 也有风险。记错的经验会污染后续任务，过期的信息会导致错误判断。所以 memory 需要写入标准、更新机制和遗忘机制。

这部分可以单独写一篇。这里先记一句：Agent memory 不是把所有聊天记录塞进上下文，而是把有价值的经验提炼出来，在需要时重新检索。

## Agent的边界

Agent 越能行动，风险越高。

普通 LLM 回答错了，最多是文本错。Agent 如果工具权限太大，可能误删文件、泄漏信息、发错请求、修改线上配置。

所以 Agent 系统一定要设计权限边界：

- 读写权限分离。
- 危险命令确认。
- 网络访问限制。
- 密钥不进上下文。
- 操作日志可追溯。
- 沙箱环境运行。

工具越强，越需要边界。Agent 的能力不是只看能做多少事，还要看能不能控制风险。

## 记录

Agent 的核心不是“会想”，而是“会循环”：

```text
理解目标 -> 选择动作 -> 调用工具 -> 观察反馈 -> 更新状态 -> 继续行动
```

ReAct 把 reasoning 和 acting 结合起来，工具调用让模型接触外部世界，observation 和状态管理让系统能持续推进任务。

我现在看 Agent，会把注意力放在三个地方：任务循环是否稳定，工具接口是否清楚，权限边界是否足够硬。模型能力当然重要，但没有好的执行框架，Agent 很容易变成会说不会做。
