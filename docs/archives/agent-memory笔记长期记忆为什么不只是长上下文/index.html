Agent Memory笔记：长期记忆为什么不只是长上下文

Agent 要长期工作，就绕不开 memory。刚开始我容易把 memory 简单理解成“把上下文变长”或者“把历史聊天都存起来”。但这样理解太粗了。

长期记忆不是把所有内容塞进模型，而是决定什么值得存、怎么存、什么时候取出来、取出来后怎么更新。

长上下文不等于长期记忆#

长上下文能解决一部分问题。比如一次任务里需要读很多文件、看完整论文、处理长日志，context window 越大越方便。

但长上下文有几个限制：

  • 成本高：每次都带大量历史，token 成本和延迟都会上升。
  • 注意力稀释：关键信息可能被埋在长上下文里。
  • 不会自动整理：历史内容还是原始记录，不是经验。
  • 不适合跨会话：每次任务都塞全部历史不现实。

Memory 关注的是跨时间的可用经验。它不是越长越好，而是越准越好。

Memory的类型#

可以先按几类理解：

短期记忆：当前任务上下文，保存正在做什么、做过什么、下一步是什么。

长期事实记忆：用户偏好、项目路径、常用命令、固定配置、个人习惯。

经验记忆：某类任务怎么做、踩过哪些坑、哪些方案失败过。

情景记忆：过去某次会话或任务的具体过程，可以被搜索和回放。

反思记忆：系统从失败或成功里总结出的原则。

不同 memory 的写入和检索方式不一样。用户偏好可以长期保存，临时日志不应该长期保存；一次失败的调试过程可以保留摘要，但没必要把所有终端输出都存进去。

写入#

Memory 的第一步是写入。写入不是所有东西都存。

值得写入的信息通常有：

  • 用户明确要求记住的偏好。
  • 项目稳定事实。
  • 重复出现的工作流程。
  • 已验证的解决方案。
  • 常见失败原因和规避方式。
  • 用户纠正过的错误假设。

不适合写入的信息：

  • 一次性临时数据。
  • 敏感密钥。
  • 未验证猜测。
  • 过期路径。
  • 大量原始日志。

写入标准很重要。如果 memory 里混入太多噪声，后面检索出来反而会误导 Agent。

压缩#

原始会话通常很长，不能直接当长期记忆。需要压缩成摘要或结构化条目。

比如一次调试过程，原始记录可能包括很多命令输出。真正值得存的是：

项目 X 的 Hugo 发布目录是 docs。
发布前需要运行 hugo。
不要提交 .DS_Store。
图片应放在 static/images/...，生成后会复制到 docs/images/...

这就是从过程到经验的压缩。

压缩时最怕丢掉条件。比如“在这个项目里”很重要，如果省略条件，经验可能被错误应用到别的项目。

检索#

Memory 只有能在正确时候取出来才有价值。

常见检索方式：

  • 关键词搜索。
  • 向量检索。
  • 按项目或用户过滤。
  • 按时间排序。
  • 根据任务类型触发。

检索结果也不能无脑塞进上下文。需要判断相关性和时效性。旧经验可能已经过期，类似任务也可能有关键差异。

我觉得比较合理的方式是：先检索候选 memory，再让 Agent 判断是否适用，并在回答或执行时显式使用。

更新和遗忘#

Memory 不能只增不减。

项目路径会变，工具命令会变，用户偏好也会变。如果旧记忆一直保留，Agent 会越来越容易被历史污染。

需要几种机制：

  • 更新：新事实覆盖旧事实。
  • 降权：旧记忆权重降低。
  • 删除：明确错误或敏感内容移除。
  • 合并：多条相似记忆整理成一条。
  • 过期：临时规则到期后不再使用。

遗忘不是缺陷，而是 memory 系统必须有的能力。没有遗忘，长期 Agent 会积累很多垃圾状态。

MemGPT#

MemGPT 是 2023 年的工作，它把 memory 管理做成类似操作系统里的内存层次。

它区分有限上下文和外部记忆，让模型通过读写操作管理长期信息。这个思路很有启发：上下文窗口像主存，外部 memory 像磁盘，模型需要决定什么时候把信息换入换出。

MemGPT 的意义不只是具体实现，而是提出一个系统视角：LLM 的上下文有限，所以 Agent 需要 memory management，而不是简单把历史拼接起来。

Generative Agents和Reflexion#

Generative Agents 强调基于记忆和反思的行为模拟。系统会保存 agent 的经历，再从经历中生成更高层的 reflections。

Reflexion 则把失败经验转成语言反馈，让 Agent 在后续尝试中避免同样错误。

这两条路线都说明一件事：memory 不只是事实存储，也可以是经验和策略存储。

对实际 Agent 来说，失败后的总结很有价值。比如“刚才测试失败是因为没有设置环境变量”，这条经验下次能直接避免重复踩坑。

和Skill的关系#

Memory 和 Skill 有重叠，但不完全一样。

Memory 更像事实和经验库，Skill 更像可执行的任务流程。

如果某条 memory 反复被用到，并且形成稳定流程，就可以沉淀成 Skill。比如一次发布经验只是 memory，多次发布后整理成 checklist，就变成 Skill。

这也是自学习 Agent 的关键闭环：

完成任务 -> 总结经验 -> 写入memory -> 重复使用 -> 固化成skill -> 后续改进skill

记录#

Agent memory 的重点是生命周期：

  • 写入：什么值得记。
  • 压缩：怎么从过程提炼经验。
  • 检索：什么时候取出来。
  • 更新：新信息怎么覆盖旧信息。
  • 遗忘：过期和错误记忆怎么清掉。

长上下文解决的是一次任务能看多少，memory 解决的是长期系统能记住什么。两者都重要，但不是一回事。
